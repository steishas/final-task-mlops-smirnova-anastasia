# Манифест разработки ML-системы

## Предыстория

Владелец агентства по продаже недвижимости из Калифорнии обратился с задачей разработки системы прогнозирования стоимости жилья на основании его характеристик. Сейчас прогнозирование выполняется вручную оценщиками, что не всегда эффективно: пропускаются значимые изменения, оценка может затягиваться на несколько дней и часто носит субъективный характер.

## Ценностное предложение

ML-сервис позволит пользователям быстро получать объективную оценку стоимости жилья, что позволит сократить время закрытия сделки и уменьшить нагрузку на оценщиков.

## Цели

- Разработать модель регрессии, которая будет предсказывать медианную стоимость жилья с ошибкой (MAPE) не более 20% на новых данных;
- Обеспечить алерт при падении бизнес-метрики в продакшене;
- Реализовать переключение между версиями модели, чтобы обеспечить откат к последней стабильной версии.

## Решение

**Модель**: LightGBM.
Выбор модели обоснован в [adr](docs/architecture/001-model-choice.md)

**Признаки**: 
 - `MedInc`        Медианный доход домохозяйств в данном квартале;
 - `HouseAge`      Медианный возраст домов в квартале;
 - `AveRooms`      Среднее количество комнат на одно домохозяйство;
 - `AveBedrms`     Среднее количество спален на одно домохозяйство;
 - `Population`    Общее количество людей, проживающих в этом квартале;
 - `AveOccup`      Среднее количество человек в одном домохозяйстве;
 - `Latitude`      Широта квартала;
 - `Longitude`     Долгота квартала.

Перед передачей в модель к признакам применяется масштабирование `StandardScaler` (встроено в `sklearn Pipeline`).

**Интеграции**: 

- `Apache Airflow` – оркестрация двух DAG:
  - `monitor_mape`: периодический (раз в час) расчёт MAPE на синтетических батчах с шумом, отправка метрики в `Prometheus`, при превышении порога — логирование нового полного набора данных в MLflow для последующего переобучения.
  - `retrain_model_manual`: ручной DAG, запускаемый инженером после алерта; выполняет обучение, регистрирует новую версию модели в MLflow Model Registry и при успехе переводит её в Production.
- `MLflow` – Tracking Server и Model Registry (стадии Staging -> Production -> Archived); используется для логирования всех экспериментов, хранения артефактов (модель, данные), управления жизненным циклом модели.
- Контейнеризация через `Docker` и `Docker Compose` (все сервисы: Airflow, MLflow, FastAPI, PostgreSQL, Prometheus, Grafana).
- `FastAPI` – REST API для инференса; поддерживает эндпоинты `/predict`, `/predict_batch`, `/health`, `/admin/reload` (перезагрузка модели из Registry), `/update_mape` (приём метрики от Airflow).
- `Prometheus` + `Grafana` – сбор метрик (MAPE, latency, количество запросов) и визуализация; настроены алерты (превышение порога MAPE 20%, latency >200ms).
- `Feature Store (PostgreSQL)` – таблицы `features` (история признаков), `scalers` (обученный StandardScaler), `predictions`, `ground_truth`.
- `GitHub Actions` – CI/CD: линтинг Python-кода (ruff), валидация `Terraform`, сборка Docker-образа `FastAPI` с упакованной моделью и деплой на Render.
- `Terraform` – IaC для развёртывания веб-сервиса в облаке `Render` (основной манифест `terraform/main.tf`).

**Ограничения**:  Построение системы в рамках учебного проекта ограничено использованием бесплатного ПО. Сервис развернут в бесплатной версии Render, старт происходит с задержкой 30-50 секунд. В облачном развёртывании модель загружается из локального файла (`USE_LOCAL_MODEL=true`), так как `MLflow Tracking Server` недоступен извне.

**Не входит в продукт**:
- Интерпретация факторов;
- Мониторинг дрейфа данных;
- A/B-тестирование для автоматического перезапуска обучения;
- Terraform destroy, поскольку сервис остается онлайн до защиты.

**Уровень зрелости ML-системы: 2**.
Система имеет:

- Версионирование кода (`Git` + `GitHub`);
- CI/CD пайплайн (`GitHub Actions`);
- `Feature Store` в `PostgreSQL`;
- Сервинг модели через `FastAPI`;
- Мониторинг качества предсказания через `Prometheus` и `Grafana`;
- Управление экспериментами через `MLflow`;
- Оркестрацию при помощи `Airflow`.

## Осуществимость

Данные доступны в открытом доступе (California Housing из sklearn). Вычислительные ресурсы: локальный компьютер / виртуальная машина с 8 ГБ RAM. Все необходимые компоненты разворачиваются через Docker Compose. IaC реализуется при помощи `Terraform` (деплой веб-сервиса на Render). Деплой в бесплатную версию облака Render.

## Данные

**Обучающий набор**: California Housing (20 640 записей), загружается через `sklearn.datasets.fetch_california_housing`.

**Продакшн данные**: имитируются скриптом мониторинга путём добавления гауссова шума к признакам и целевой переменной исходной выборки. Сгенерированные датасеты логируются как артефакты в MLflow.

**Разметка**: данные уже размечены (целевая переменная `MedHouseVal`). Для демонстрации разметка создаётся тем же скриптом, который добавляет шум к таргету.

**Feature Store**: таблица `features` в PostgreSQL хранит предобработанные признаки и `scalers` — сериализованный объект StandardScaler. Обновляется при каждом запуске обучения (Airflow). Все версии сырых и обработанных данных отслеживаются через MLflow Artifacts, что гарантирует воспроизводимость обучения.

## Метрики

### Бизнес-метрики

| SLI | Описание | SLO |
|-----|----------|-----|
| MAPE | Средняя абсолютная процентная ошибка предсказания стоимости | < 20% |

**Действие при нарушении**: алерт в `Grafana`, при статистически значимом ухудшении ручной запуск переобучения модели (DAG `retrain_model_manual`).

---

### Модельные метрики

| SLI | Описание | SLO |
|-----|----------|-----|
| $R^2$ | Коэффициент детерминации | > 0.80 |

**Измерение**: вычисляется при каждом переобучении и логируется в MLflow. При падении ниже порога новая версия модели не продвигается в Production.

---

### Технические метрики

| SLI | Описание | SLO |
|-----|----------|-----|
| Latency p95 | 95-й перцентиль времени ответа `API` | < 200 мс |

**Измерение**: Prometheus собирает метрику из `FastAPI`.

## Оценка качества модели

**Офлайн-метрики**: MAPE и $R^2$ вычисляются на отложенной выборке и логируются в MLflow.

**Онлайн-метрики**: MAPE обновляется через Airflow, latency собирается Prometheus из FastAPI. Обе метрики визуализируются в Grafana, при превышении порогов срабатывают алерты.

## Подбор модели

В качестве baseline использовалась линейная регрессия с L2-регуляризацией. Далее последовательно опробованы RandomForestRegressor и LightGBM. Лучшая модель выбиралась по значению MAPE при приемлемой сложности. Все эксперименты зарегистрированы в `MLflow` (ноутбуки и скрипты в папке `model-experiments`), артефакты сохранены, код версионирован.

## Инференс

Инференс в реальном времени через `REST API (FastAPI)`. Обучение запускается **вручную** инженером после срабатывания алерта MAPE (DAG `retrain_model_manual`). Мониторинг качества выполняется автоматически по расписанию (DAG `monitor_mape`).

FastAPI-сервис при старте загружает модель со стадией `Production` из `MLflow` (в локальной среде) или из локального файла (в облаке) и может перезагрузить её по эндпоинту `/admin/reload`. Это обеспечивает моментальный откат к предыдущей версии (перевод старой модели обратно в Production через MLflow) и исключает простой.

## Обратная связь

- Явная: реальные цены продаж (эмулируются через зашумлённый таргет), поступающие с задержкой, для расчёта MAPE.

MDD для принятия решений: при превышении порога MAPE отправляется алерт. После проверки (визуальной или статистической) инженер вручную запускает переобучение. В будущем может быть добавлен автоматический статистический тест значимости ухудшения.
